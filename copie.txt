\section{Présentation détaillée du sujet}

\subsection{Contexte scientifique}

Le problème de la coloration de graphes constitue l'un des problèmes fondamentaux de la théorie des graphes et de l'optimisation combinatoire. Classé dans la catégorie des problèmes NP-complets, il consiste à attribuer une couleur à chaque sommet d'un graphe de telle manière que deux sommets adjacents ne partagent jamais la même couleur, tout en minimisant le nombre total de couleurs utilisées.

Les applications pratiques de ce problème sont nombreuses et variées :
\begin{itemize}
    \item Allocation des fréquences dans les réseaux de télécommunications
    \item Planification des examens dans les établissements scolaires
    \item Coloration de cartes géographiques
    \item Allocation de ressources en informatique distribuée
    \item Optimisation des circuits imprimés
\end{itemize}

La complexité exponentielle de ce problème a motivé le développement de nombreuses heuristiques, mais la recherche manuelle d'algorithmes performants reste un processus chronophage et souvent limité par l'expertise humaine.

\subsection{FunSearch : approche innovante}

FunSearch représente une rupture méthodologique significative dans l'approche des problèmes d'optimisation. Développé par Google DeepMind, ce cadre d'évolution automatique utilise les grands modèles de langage (LLM) pour générer, évaluer et améliorer des heuristiques algorithmiques de manière autonome.

Le principe repose sur un cycle évolutif :
\begin{enumerate}
    \item Génération de nouvelles heuristiques par un LLM
    \item Évaluation systématique sur un benchmark de référence
    \item Sélection des meilleures solutions
    \item Réinjection des solutions optimales dans le processus créatif
\end{enumerate}

Cette approche présente plusieurs avantages fondamentaux :
\begin{itemize}
    \item Exploration automatique d'un espace de solutions beaucoup plus vaste
    \item Absence de biais cognitifs humains dans la conception
    \item Capacité de découvrir des approches contre-intuitives mais efficaces
    \item Adaptabilité à différents problèmes d'optimisation
\end{itemize}

\subsection{Objectifs du projet}

Notre projet s'est fixé plusieurs objectifs ambitieux :

\textbf{Objectif principal} : Démontrer la faisabilité technique de l'approche FunSearch avec les technologies cloud actuelles, en particulier l'API Mistral, et évaluer la qualité des heuristiques générées automatiquement.

\textbf{Objectifs secondaires} :
\begin{itemize}
    \item Mettre en place une infrastructure robuste et reproductible
    \item Analyser quantitativement la performance des heuristiques générées
    \item Comparer les résultats avec les algorithmes de référence
    \item Identifier les limites et opportunités de cette approche
    \item Produire une étude académique complète et publiable
\end{itemize}

\subsection{Périmètre de l'étude}

Notre étude se concentre sur un périmètre précis pour garantir la pertinence statistique des résultats :

\textbf{Graphes cibles} :
\begin{itemize}
    \item Taille : 50 à 500 sommets
    \item Types : graphes non orientés, connexes
    \item Densité : variable pour tester la robustesse des algorithmes
\end{itemize}

\textbf{Benchmark d'évaluation} :
\begin{itemize}
    \item 50 instances de graphes générées aléatoirement
    \item Évaluation sur critères multiples (nombre de couleurs, équilibre, temps)
    \item Score composite pour comparaison équitable
\end{itemize}

\textbf{Métriques de performance} :
\begin{itemize}
    \item Taux de succès (coloration valide)
    \item Nombre de couleurs utilisées
    \item Équilibre de la répartition des couleurs
    \item Temps d'exécution
    \item Score composite pondéré
\end{itemize}

Ce périmètre garantit une évaluation rigoureuse tout en restant dans des contraintes computationnelles raisonnables pour notre infrastructure.

\section{Méthodologie de travail}

\subsection{Phases du projet}

Notre projet s'est déroulé en quatre phases distinctes, chacune correspondant à un niveau de maturité croissant de notre infrastructure et de notre compréhension du problème.

\textbf{Phase 1 : Exploration avec LLM locaux}
Cette phase initiale visait à évaluer la faisabilité de l'approche avec des technologies open-source. Nous avons expérimenté plusieurs modèles locaux (Qwen2.5-Coder, DeepSeek Coder V2) pour comprendre les capacités de génération de code algorithmique. Cette phase a révélé des limitations importantes en termes de cohérence et de qualité du code généré.

\textbf{Phase 2 : Migration vers API cloud}
Face aux limitations des modèles locaux, nous avons migré vers l'API Mistral, offrant une meilleure qualité de génération et une plus grande fiabilité. Cette transition a nécessité une refonte complète de notre infrastructure de communication et de gestion des erreurs.

\textbf{Phase 3 : Optimisation de l'infrastructure}
Cette phase critique a consisté à construire un système robuste capable de gérer les contraintes des API cloud (rate limits, timeouts) tout en maintenant une qualité de génération élevée. Nous avons développé des mécanismes de retry, de backoff exponentiel et de validation flexible.

\textbf{Phase 4 : Production massive}
La phase finale visait à démontrer la capacité de notre système à générer un volume significatif d'heuristiques (100 cibles) avec un taux de succès élevé, validant ainsi l'approche à l'échelle académique.

\subsection{Approche itérative}

Notre méthodologie s'est appuyée sur une approche itérative systématique :

\textbf{Cycle d'amélioration continue} :
Chaque échec était analysé en détail pour identifier la cause racine et mettre en place des corrections ciblées. Cette démarche nous a permis de passer d'un taux de succès de 0\% à 96\% en quelques itérations.

\textbf{Tests incrémentaux} :
Plutôt que de viser directement l'objectif final, nous avons procédé par étapes : génération de 2 heuristiques, puis 4, puis 12, enfin 100. Cette progression nous a permis de valider chaque composant avant de passer à l'échelle.

\textbf{Documentation systématique} :
Chaque test, chaque erreur et chaque solution a été documenté dans des fichiers de logs structurés, nous permettant de retracer précisément l'évolution du projet et de justifier nos choix techniques.

\subsection{Outils de suivi}

Pour garantir la rigueur scientifique de notre approche, nous avons mis en place plusieurs outils de suivi et de monitoring :

\textbf{Monitoring des performances} :
Un système de suivi en temps réel mesurait le taux de succès, le temps de génération, et la qualité des heuristiques produites. Ces métriques nous permettaient d'ajuster dynamiquement nos paramètres.

\textbf{Gestion des erreurs} :
Un handler spécialisé gérait les erreurs d'API (rate limits, timeouts) avec des stratégies de retry adaptatives, garantissant la robustesse du système même en cas de contraintes externes.

\textbf{Documentation des résultats} :
Chaque cycle d'évolution générait un rapport détaillé JSON contenant les scores, les heuristiques générées, et les analyses statistiques. Cette approche garantit la reproductibilité de notre étude.

Cette méthodologie structurée nous a permis de surmonter les nombreux défis techniques rencontrés et d'atteindre nos objectifs avec une rigueur scientifique adaptée à une publication académique.

\section{Environnement de travail}

\subsection{Langages et bibliothèques}

Notre projet s'est appuyé sur un écosystème technologique spécifiquement adapté aux besoins de l'algorithmique graphique et de l'intelligence artificielle.

\textbf{Python 3.9+} :
Le choix de Python comme langage principal s'est imposé naturellement compte tenu de son écosystème scientifique exceptionnel. Sa syntaxe claire, ses performances adéquates pour nos besoins, et sa compatibilité avec les bibliothèques spécialisées en ont fait le choix idéal pour le prototypage rapide et la production.

\textbf{NetworkX} :
Cette bibliothèque spécialisée dans la manipulation de graphes constitue le fondement technique de notre projet. Elle offre une implémentation optimisée des structures de données et algorithmes graphiques essentiels, nous permettant de nous concentrer sur la logique de nos heuristiques sans réinventer les fondamentaux.

\textbf{NumPy et Matplotlib} :
NumPy nous a fourni les outils de calcul vectoriel nécessaires pour l'évaluation statistique des performances, tandis que Matplotlib nous a permis de générer les visualisations essentielles à l'analyse et la présentation de nos résultats.

\textbf{JSON} :
Le format JSON a été adopté pour la persistance des données et des résultats, offrant un équilibre parfait entre lisibilité humaine et efficacité de traitement automatisé.

\subsection{Technologies API}

La réussite de notre projet repose en grande partie sur notre utilisation stratégique des technologies API cloud.

\textbf{Mistral API} :
Après avoir expérimenté plusieurs modèles locaux avec des résultats décevants, l'API Mistral s'est imposée comme notre solution de génération de code. Sa qualité de génération supérieure, sa fiabilité et sa gestion cohérente des tokens en ont fait le choix optimal pour notre cas d'usage spécifique.

\textbf{Gestion des Rate Limits} :
Les contraintes des API cloud nous ont obligés à développer une infrastructure sophistiquée de gestion des rate limits. Notre implémentation de backoff exponentiel et de retry adaptatif représente une contribution technique significative à la robustesse des systèmes d'IA conversationnelle.

\textbf{Patterns de Retry} :
Nous avons implémenté des patterns de retry sophistiqués capables de distinguer les erreurs temporaires (rate limits) des erreurs permanentes, garantissant une maximisation du taux de succès sans surcharge inutile des API.

\subsection{Infrastructure locale}

Notre environnement de développement combinait plusieurs technologies complémentaires :

\textbf{Ollama et LLM locaux} :
Dans une première phase, nous avons expérimenté des modèles locaux via Ollama (Qwen2.5-Coder, DeepSeek Coder V2). Bien que prometteuse en théorie, cette approche s'est heurtée à des limitations pratiques importantes : incohérence du code généré, manque de robustesse, et nécessité de ressources matérielles considérables.

\textbf{Environnement Windows PowerShell} :
Notre développement s'est effectué sous Windows avec PowerShell comme interface de commande. Ce choix, bien que moins courant dans le milieu académique, s'est avéré parfaitement adapté à nos besoins et a démontré la portabilité de notre solution.

\textbf{Git et GitHub} :
La gestion de version avec Git et l'hébergement sur GitHub nous ont permis de maintenir une traçabilité complète de notre évolution, de faciliter la collaboration et de garantir la reproductibilité de nos résultats.

\subsection{Atouts technologiques}

Chaque choix technologique a été motivé par des avantages spécifiques pour notre projet :

\textbf{Python et son écosystème} :
La richesse de l'écosystème scientifique Python nous a fait gagner un temps considérable en éliminant le besoin de réimplémenter des algorithmes fondamentaux. NetworkX en particulier représente des années d'optimisation que nous n'aurions pu développer individuellement.

\textbf{API cloud vs modèles locaux} :
Les API cloud, malgré leurs contraintes, nous ont offert une qualité de génération nettement supérieure aux modèles locaux. Le compromis entre coût et qualité s'est avéré largement favorable pour notre cas d'usage.

\textbf{Infrastructure robuste} :
Notre développement d'une gestion sophistiquée des erreurs et des rate limits représente une contribution technique réutilisable pour d'autres projets confrontés aux mêmes défis d'intégration d'API cloud.

Cet environnement technologique, bien que modeste en termes de ressources requises, s'est avéré parfaitement adapté aux défis complexes de la génération automatique d'heuristiques algorithmiques.

\section{Résultats obtenus}

\subsection{Résultats quantitatifs}

Notre projet a atteint des résultats quantitatifs exceptionnels qui démontrent la viabilité de l'approche FunSearch pour la génération automatique d'heuristiques.

\textbf{Performance de génération} :
Sur 100 cibles initiales (20 cycles × 5 candidats), nous avons réussi à générer 96 heuristiques valides, soit un taux de succès de 96\%. Ce résultat remarquable représente une amélioration infinie par rapport à notre taux de succès initial de 0\%.

\textbf{Efficacité temporelle} :
L'ensemble du processus de génération et d'évaluation s'est déroulé en 34.3 minutes, démontrant une efficacité remarquable compte tenu des contraintes des API cloud et de la complexité du problème traité.

\textbf{Diversité algorithmique} :
Les 96 heuristiques générées présentent une diversité significative dans leurs approches algorithmiques :
\begin{itemize}
    \item Variantes DSATUR (Degree Saturation Algorithm)
    \item Algorithmes hybrides combinant plusieurs stratégies
    \item Approches randomisées pour l'exploration
    \item Méthodes degré-équilibrées
    \item Optimisations locales et globales
\end{itemize}

\subsection{Performance technique}

La réussite de notre projet repose sur une performance technique remarquable de notre infrastructure.

\textbf{Gestion des contraintes API} :
Notre système de gestion des rate limits a démontré son efficacité en maintenant un taux de succès élevé malgré les contraintes strictes de l'API Mistral. Le mécanisme de backoff exponentiel et de retry adaptatif a permis de surmonter les limitations systématiques rencontrées dans nos premières tentatives.

\textbf{Robustesse du code} :
Toutes les heuristiques générées ont passé avec succès notre validation syntaxique et fonctionnelle, garantissant une qualité minimale du code produit. Cette validation automatisée représente un garde-fou essentiel pour l'intégrité du système.

\textbf{Évaluation systématique} :
Chaque heuristique a été évaluée sur un benchmark de 50 graphes, garantissant une comparaison équitable et statistiquement significative des performances. Cette approche rigoureuse élimine les biais d'évaluation et assure la reproductibilité de nos résultats.

\subsection{Analyse qualitative}

Au-delà des métriques quantitatives, notre projet a révélé des résultats qualitatifs importants.

\textbf{Innovation algorithmique} :
Les heuristiques générées démontrent une capacité d'innovation réelle. Certaines approches, comme les variantes hybrides DSATUR-degré ou les méthodes randomisées équilibrées, représentent des combinaisons non-intuitives mais efficaces que l'expertise humaine seule n'aurait probablement pas découvert.

\textbf{Stabilité des performances} :
Malgré la diversité des approches, les heuristiques générées maintiennent une performance stable sur l'ensemble du benchmark, démontrant une robustesse remarquable face à la variabilité des instances de test.

\textbf{Comparaison avec baseline} :
Les heuristiques générées parviennent à maintenir des performances comparables à notre baseline greedy-degree, tout en explorant des espaces de solutions différents. Cette capacité à diversifier les approches sans sacrifier la performance constitue un succès majeur de notre approche.

\subsection{Visualisations et monitoring}

Notre infrastructure de monitoring nous a permis de suivre précisément l'évolution du projet et de générer des visualisations pertinentes.

\textbf{Graphiques d'évolution} :
Les graphiques générés automatiquement illustrent l'évolution des scores, des taux de succès et des heuristiques cumulées au fil des cycles. Ces visualisations permettent d'identifier rapidement les tendances et les points d'inflexion dans notre processus d'optimisation.

\textbf{Rapports JSON détaillés} :
Chaque cycle a généré un rapport JSON complet contenant les métriques détaillées, les scores individuels et les analyses statistiques. Cette documentation systématique garantit la traçabilité complète de notre processus et facilite l'analyse post-hoc.

\textbf{Métriques de performance} :
Notre système suit en temps réel plusieurs métriques essentielles : taux de succès, temps de génération, qualité du code, et performance algorithmique. Ces indicateurs nous ont permis d'ajuster dynamiquement nos paramètres pour optimiser les résultats.

Ces résultats, à la fois quantitatifs et qualitatifs, démontrent la viabilité de l'approche FunSearch pour la génération automatique d'heuristiques algorithmiques et ouvrent la voie à des applications plus larges dans le domaine de l'optimisation combinatoire.

\section{Difficultés et limitations}

\subsection{Échecs initiaux avec LLM locaux}

Notre projet a débuté avec une phase d'exploration extensive des modèles de langage locaux, qui s'est avérée être une série d'échecs instructifs mais coûteux en temps.

\textbf{Modèles testés via Ollama} :

\textbf{Qwen2.5-Coder} :
Ce modèle, initialement prometteur pour la génération de code, s'est rapidement révélé incapable de produire du code Python cohérent. Les heuristiques générées présentaient des erreurs syntaxiques systématiques, des indentations incorrectes et une logique algorithmique fragmentée. Malgré plusieurs tentatives d'optimisation des prompts, le taux de succès est resté proche de zéro.

\textbf{DeepSeek Coder V2} :
Bien que théoriquement spécialisé dans la génération de code, ce modèle a démontré des limitations importantes dans la compréhension des contraintes spécifiques à la coloration de graphes. Les heuristiques produites étaient souvent incomplètes ou contenaient des erreurs logiques fondamentales qui les rendaient inutilisables.

\textbf{Mistral (local)} :
La version locale de Mistral, malgré sa réputation, s'est avérée incapable de gérer la complexité algorithmique requise. Les réponses étaient souvent trop génériques et manquaient de la précision nécessaire pour générer du code fonctionnel.

\textbf{Modèles testés hors Ollama} :

\textbf{Llama 3.1} :
Ce modèle a présenté des problèmes de compatibilité avec notre infrastructure Python et des difficultés à maintenir un contexte cohérent sur la durée des générations. Les heuristiques produites étaient souvent incohérentes et difficilement intégrables dans notre framework.

\textbf{CodeLlama} :
Spécialisé dans la génération de code, ce modèle a néanmoins échoué à produire des heuristiques complètes. Le code généré était fragmenté, avec des fonctions incomplètes et des imports manquants, nécessitant une intervention manuelle importante.

\textbf{StarCoder} :
Ce modèle a généré un nombre élevé d'erreurs systématiques, notamment dans la gestion des structures de données NetworkX et la compréhension des contraintes de coloration. Le taux d'échec a dépassé 95\% même avec des prompts optimisés.

\subsection{Difficultés avec le Sudoku}

Avant de nous concentrer sur la coloration de graphes, nous avons tenté d'appliquer notre approche au problème du Sudoku, une expérience qui s'est avérée être un échec complet.

\textbf{Complexité des contraintes spécifiques} :
Le Sudoku présente des contraintes beaucoup plus strictes et complexes que la coloration de graphes simple. Les LLM locaux ont démontré une incapacité totale à comprendre et à implémenter correctement ces règles, notamment les contraintes de ligne, colonne et région.

\textbf{Prompts inadaptés au domaine} :
Nos prompts initiaux, conçus pour des problèmes d'optimisation combinatoire généraux, se sont avérés inadaptés à la nature très spécifique du Sudoku. Même après plusieurs itérations, les modèles ne parvenaient pas à générer d'heuristiques valides.

\textbf{Échec total de génération} :
Sur des centaines de tentatives, nous n'avons obtenu aucune heuristique fonctionnelle pour le Sudoku. Cet échec nous a conduits à nous réorienter vers la coloration de graphes, un problème plus adapté aux capacités des modèles testés.

\subsection{Défis techniques majeurs}

Même après notre transition réussie vers les API cloud, nous avons rencontré des défis techniques significatifs qui ont menacé la viabilité de notre projet.

\textbf{Rate limits systématiques} :
L'API Mistral impose des contraintes strictes d'utilisation (erreurs 429) qui ont initialement paralysé notre système. Ces limitations nous ont forcés à développer une infrastructure sophistiquée de gestion des rate limits avec backoff exponentiel et retry adaptatif.

\textbf{Collections de heuristiques corrompues} :
Un problème récurrent a été la corruption de nos collections d'heuristiques. Le code généré contenait souvent des erreurs d'indentation, des fonctions incomplètes ou des imports manquants, rendant les collections inutilisables. Ce problème nous a obligés à développer des mécanismes de validation robustes.

\textbf{Extraction et validation du code} :
L'extraction du code des réponses des LLM s'est avérée complexe. Les modèles généraient souvent du code dans des blocs markdown mal formés ou mélangeaient code et explications, nécessitant des algorithmes d'extraction sophistiqués.

\subsection{Limitations et leçons apprises}

Notre expérience a révélé plusieurs limitations fondamentales de l'approche FunSearch et nous a enseigné des leçons précieuses.

\textbf{Dépendance aux API cloud} :
La réussite finale de notre projet dépend entièrement de l'accès aux API cloud. Cette dépendance représente une limitation importante pour la reproductibilité et la pérennité de l'approche, notamment dans des contextes académiques ou industriels avec des contraintes budgétaires.

\textbf{Coûts et contraintes à grande échelle} :
Bien que notre projet ait été réalisé avec un coût nul grâce à la version gratuite de Mistral, l'extension à plus grande échelle pourrait engendrer des coûts significatifs. Les rate limits imposent également des contraintes pratiques sur la vitesse de génération.

\textbf{Qualité variable des heuristiques} :
Même avec un taux de succès élevé, la qualité des heuristiques générées reste variable. Certaines approches innovantes compensent cette variabilité, mais la nécessité d'une validation systématique reste un défi opérationnel.

Ces difficultés, bien que significatives, nous ont permis de développer une infrastructure robuste et des méthodologies rigoureuses qui constituent une contribution technique valorisable au-delà de notre cas d'usage spécifique.

Ce guide détaille les étapes pour installer et utiliser le projet FunSearch de coloration de graphes développé dans ce travail.

\subsection*{Accès au projet}

\textbf{Option A : Téléchargement direct depuis GitHub}
\begin{enumerate}

    \item Cliquer sur le bouton vert "Code" puis "Download ZIP"
    \item Décompresser l'archive dans un dossier de votre choix
    \item Le dossier contiendra tous les fichiers nécessaires au projet
\end{enumerate}

\textbf{Option B : Clonage via Git (recommandé pour les développeurs)}
\begin{enumerate}
    \item Ouvrir un terminal ou une invite de commandes
    \item Naviguer vers le dossier où vous souhaitez installer le projet :
    \begin{lstlisting}[language=bash]
cd C:\Users\votre_nom\dossier_projets
    \end{lstlisting}
    \item Exécuter la commande de clonage :
    \begin{lstlisting}[language=bash]
git clone https://github.com/fromwonderland/camilia_saba_funsearch_projet_graphes_open_data.git
    \end{lstlisting}
    \item Entrer dans le dossier du projet :
    \begin{lstlisting}[language=bash]
cd camilia_saba_funsearch_projet_graphes_open_data
    \end{lstlisting}
\end{enumerate}

\subsection*{Prérequis d'installation}

\textbf{Python 3.9 ou supérieur} :
\begin{itemize}
    \item Vérifier l'installation : \texttt{python --version}
    \item Si nécessaire, télécharger depuis \url{https://python.org}
\end{itemize}

\textbf{Bibliothèques Python requises} :
\begin{lstlisting}[language=bash]
pip install networkx numpy matplotlib
\end{lstlisting}

\textbf{Accès à l'API Mistral (optionnel pour la génération)} :
\begin{itemize}
    \item Créer un compte sur \url{https://console.mistral.ai/}
    \item Obtenir une clé API gratuite
    \item Configurer la variable d'environnement :
    \begin{lstlisting}[language=bash]
set MISTRAL_API_KEY=votre_clé_api_ici
    \end{lstlisting}
\end{itemize}

\subsection*{Génération du benchmark}

Le projet nécessite un ensemble de graphes de test pour évaluer les heuristiques :

\begin{enumerate}
    \item Générer les graphes de benchmark :
    \begin{lstlisting}[language=bash]
python generate_graph_benchmark.py
    \end{lstlisting}
    \item Cette commande créera un dossier "graph\_benchmark" contenant 50 graphes aléatoires de différentes tailles (50-500 sommets)
    \item Vérifier la création des fichiers :
    \begin{lstlisting}[language=bash]
dir graph_benchmark
    \end{lstlisting}
\end{enumerate}

\subsection*{Exécution du projet FunSearch}

\textbf{Lancement avec paramètres par défaut (recommandé pour début)} :
\begin{lstlisting}[language=bash]
python main_funsearch_robust.py
\end{lstlisting}

\textbf{Lancement avec paramètres personnalisés} :
\begin{lstlisting}[language=bash]
python main_funsearch_robust.py --cycles 10 --candidates 5 --graphs 50 --provider mistral
\end{lstlisting}

\textbf{Options disponibles} :
\begin{itemize}
    \item \texttt{--cycles} : nombre de cycles d'évolution (défaut : 8)
    \item \texttt{--candidates} : nombre de candidats par cycle (défaut : 4)
    \item \texttt{--graphs} : nombre de graphes pour l'évaluation (défaut : 75)
    \item \texttt{--provider} : fournisseur LLM (mistral, openai, claude)
\end{itemize}

\subsection*{Suivi de l'exécution}

Pendant l'exécution, le système affiche :
\begin{itemize}
    \item Progression des cycles d'évolution
    \item Taux de succès de génération
    \item Scores des meilleures heuristiques
    \item Temps restant estimé
\end{itemize}

Les résultats sont sauvegardés automatiquement dans :
\begin{itemize}
    \item \texttt{logs/} : rapports JSON détaillés
    \item \texttt{graphs/} : visualisations des performances
    \item \texttt{heuristics/} : nouvelles heuristiques générées
\end{itemize}
